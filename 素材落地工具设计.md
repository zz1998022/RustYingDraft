# 素材落地工具设计

## 1. 目标

这个工具的目标很单一：

**把用户下载到本地的“素材 + 项目描述”转换成当前机器可打开的剪映草稿。**

它不负责：

- 在线编辑
- 云端同步
- 自动导出视频
- 替代 Web 主产品

它只负责最后一公里：

1. 让用户选择项目包或素材目录
2. 让用户选择剪映草稿箱位置
3. 把素材路径重写成当前机器上的绝对路径
4. 生成最终可打开的剪映草稿

## 2. 为什么需要这个工具

剪映草稿有一个核心限制：

- `draft_content.json` 中引用的素材路径必须是**本机绝对路径**

这意味着：

- 服务端生成的路径对用户电脑通常无效
- 单纯下载 `draft_content.json + assets` 不能保证可直接打开

所以必须在用户本机做一次“落地”：

1. 确定素材真实存放位置
2. 确定剪映草稿箱位置
3. 重写草稿中的素材路径
4. 输出最终草稿

## 3. 产品定位

建议把它做成一个独立的 Companion App，而不是改造主 Web 项目。

推荐定位：

- 名称可暂定为：`剪映草稿导入器` / `草稿落地工具`
- 技术形态：跨平台桌面应用
- 主要平台：Windows、macOS
- 交互形式：图形化

## 4. 推荐技术方案

推荐组合：

- 桌面壳：`Tauri`
- 前端：`Vue + Vite`
- 本地逻辑：`Rust`
- 草稿内核：复用当前 `YingDraft`

原因：

1. 你现有前端栈可复用
2. Rust 草稿生成逻辑已经有基础
3. Tauri 适合做文件系统强相关的小工具
4. 跨平台分发成本比 Electron 更低

## 5. 用户操作流程

第一版建议采用 4 步流程。

### 5.1 第一步：选择项目来源

支持两种来源：

1. 选择项目包 `.zip`
2. 选择已解压的项目目录

项目包里建议至少包含：

- `bundle.json`
- `timeline.json` 或 `manifest.json`
- `assets/`

### 5.2 第二步：确认素材目录

用户需要看到：

- 当前素材目录
- 素材数量
- 缺失素材提示

如果项目包中素材不全，可以允许用户手动补选目录。

### 5.3 第三步：选择剪映草稿箱目录

这里必须图形化。

界面要求：

- 自动尝试检测默认剪映草稿目录
- 如果检测到，自动填入
- 用户可以点击“更换目录”
- 用户可以点击“打开目录”

说明文案要清楚：

- 这个目录是剪映保存草稿的目录
- 工具会在这里创建一个新的草稿文件夹

### 5.4 第四步：生成草稿

用户点击“生成草稿”后，程序执行：

1. 校验素材
2. 解析项目描述
3. 解析剪映草稿箱目录
4. 建立素材绝对路径映射
5. 生成草稿目录
6. 输出成功或失败结果

## 6. UI 页面建议

第一版建议只做 4 个页面。

### 6.1 首页

显示：

- 选择项目包按钮
- 选择项目目录按钮
- 最近使用记录
- 设置入口

### 6.2 导入确认页

显示：

- 项目名称
- 素材数量
- 素材目录
- 剪映草稿箱目录
- 输出草稿名称
- 开始生成按钮

### 6.3 进度页

显示：

- 正在校验项目
- 正在解析素材
- 正在生成绝对路径
- 正在写入草稿
- 完成百分比

### 6.4 结果页

显示：

- 成功 / 失败
- 草稿目录
- 剪映草稿箱目录
- 打开目录按钮
- 复制日志按钮

## 7. 输入形式设计

工具建议支持两种输入模式。

### 7.1 模式 A：项目包模式

项目包中包含：

- `bundle.json`
- `timeline.json` 或 `manifest.json`
- `assets/`

优点：

- 导入流程标准化
- 对用户最简单
- 适合长期产品化

### 7.2 模式 B：素材目录模式

用户直接选择一个本地目录，目录中可能包含：

- 视频
- 音频
- 图片
- 字幕
- 一个项目描述 JSON

优点：

- 对当前阶段更灵活
- 不要求服务端先压成统一 zip

建议第一版同时支持 A 和 B，这样前期更好过渡。

## 8. 输出形式设计

输出目标是：

**在用户选择的剪映草稿箱目录下创建一个新的草稿目录。**

例如用户选择：

```text
D:\JianyingPro Drafts
```

工具最终生成：

```text
D:\JianyingPro Drafts\my_project_20260422\
  draft_content.json
  draft_meta_info.json
```

如果你希望更稳，也可以先生成到应用工作目录，再复制到草稿箱目录。

## 9. 本地目录策略

建议应用自己维护一套工作目录，用于缓存解压内容和素材绝对路径。

Windows 示例：

```text
C:\Users\<user>\AppData\Local\YourApp\
  projects\
  cache\
  logs\
```

macOS 示例：

```text
~/Library/Application Support/YourApp/
  projects/
  cache/
  logs/
```

每个项目建议使用独立目录：

```text
projects/<project_id>/
  bundle/
  assets/
  generated_draft/
  logs/
```

这样做的好处：

1. 绝对路径稳定
2. 容易排查问题
3. 方便缓存和重复打开
4. 后面容易做增量更新

## 10. 剪映草稿箱目录策略

这部分要尽量做得稳，不要依赖单一路径猜测。

建议策略：

1. 程序启动时尝试检测常见草稿目录
2. 如果检测到，作为默认值显示
3. 始终允许用户手动选择
4. 记住上次成功使用的目录

交互上建议加两个按钮：

- `选择目录`
- `恢复默认检测结果`

## 11. 核心处理流程

这个工具真正要做的事情可以拆成 6 步。

### 11.1 读取项目描述

根据输入模式读取：

- `bundle.json`
- `timeline.json`
- 或简化版 `manifest.json`

### 11.2 解析素材

对每个素材建立记录：

- `asset_id`
- 素材类型
- 原始来源
- 当前本地路径

### 11.3 规范化成本机绝对路径

这是核心步骤。

目标是得到：

```text
asset_id -> 本机绝对路径
```

例如：

```text
video_main -> D:\Users\xxx\AppData\Local\YourApp\projects\proj_1\assets\main.mp4
```

### 11.4 映射到内部 `Project`

把上层 timeline 或 manifest 转成：

- `jy_schema::Project`

### 11.5 生成草稿

调用：

- `jy_draft::writer::write_draft`

输出到临时目录或目标目录。

### 11.6 复制到剪映草稿箱

如果前一步生成在工作目录，这一步再复制到用户选择的剪映草稿箱位置。

## 12. 需要暴露给前端的核心能力

如果用 Tauri，前后端通信建议只保留少量稳定接口。

### 12.1 `pick_project_source`

用途：

- 选择 zip 或项目目录

### 12.2 `detect_draft_box_dir`

用途：

- 自动检测剪映草稿箱目录

### 12.3 `pick_draft_box_dir`

用途：

- 手工选择剪映草稿箱目录

### 12.4 `inspect_project_bundle`

用途：

- 读取项目名
- 统计素材数量
- 给前端展示确认信息

### 12.5 `generate_draft_from_bundle`

输入：

- 项目源路径
- 草稿箱目录
- 输出草稿名

输出：

- 成功 / 失败
- 最终草稿目录
- 日志信息

## 13. 建议的 Rust 模块拆分

建议在 companion app 中继续拆分成这几层。

### 13.1 `bundle`

负责：

- 读取 zip
- 解压项目包
- 解析 `bundle.json`

### 13.2 `timeline_import`

负责：

- 解析 `timeline.json`
- 做版本兼容

### 13.3 `asset_resolver`

负责：

- 校验素材存在
- 复制素材到工作目录
- 下载缺失远程资源
- 生成绝对路径映射

### 13.4 `draft_box`

负责：

- 检测剪映草稿箱目录
- 校验目录是否可写
- 生成目标草稿路径

### 13.5 `project_mapper`

负责：

- 将上层时间轴映射成 `jy_schema::Project`

### 13.6 `draft_output`

负责：

- 调用 `write_draft`
- 复制草稿
- 返回结果

## 14. 错误处理设计

建议把错误分成下面几类，让 UI 能给用户更清楚的提示。

### 14.1 项目错误

- 项目包损坏
- 缺少 `bundle.json`
- 缺少 `timeline.json`
- 版本不兼容

### 14.2 素材错误

- 素材缺失
- 素材格式不支持
- 素材路径不存在
- 字幕文件损坏

### 14.3 草稿箱错误

- 草稿箱目录不存在
- 草稿箱目录不可写
- 用户没有权限

### 14.4 生成错误

- JSON 转换失败
- 草稿写入失败
- 目标目录重名冲突

## 15. 最小 MVP 范围

第一版建议只做下面这些能力：

1. 选择项目包或项目目录
2. 选择剪映草稿箱目录
3. 检测并整理本地素材
4. 把素材转成绝对路径
5. 生成视频、音频、文字、图片四类片段
6. 输出最终剪映草稿
7. 图形化显示成功 / 失败结果

先不做：

- 自动打开剪映
- 自动导出
- 复杂模板编辑
- 账号系统
- 云同步

## 16. 实现优先级建议

如果你明天要开始做，我建议按这个顺序推进：

1. 先定义项目包格式
2. 先做 Rust CLI 原型
3. 跑通“项目包 -> 绝对路径 -> 剪映草稿”
4. 再把 CLI 能力挂到 Tauri GUI
5. 最后再做草稿箱自动检测和最近使用记录

这样做的原因是：

- 先跑通数据链路
- 再做图形层
- 风险最小
