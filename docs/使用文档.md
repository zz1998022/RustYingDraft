# YingDraft 使用文档

## 1. 项目定位

`YingDraft` 是一个用 Rust 生成和编辑剪映草稿的工程。

当前已经具备三类核心能力：

1. 通过统一的 `Project / Track / Clip` 数据结构直接生成剪映草稿
2. 将现有剪映草稿作为模板进行素材替换、文本替换和草稿复制
3. 将阿里云 VOD 导出的时间轴 JSON 转换为剪映草稿，并自动下载远程素材到本地

这个工程的目标不是模拟剪映 GUI，而是直接生成剪映草稿目录中的：

- `draft_content.json`
- `draft_info.json`
- `draft_meta_info.json`

这样你可以把 Rust 当成“草稿生成内核”，再往上接：

- Web 前端
- Tauri 桌面端
- 批处理 CLI
- AI 视频生成流水线

## 2. 工作区结构

项目采用 Rust workspace，核心 crate 如下：

- `crates/jy_schema`
  - 定义统一数据模型，包括 `Project`、`Track`、`Clip`、`Material`、`TextStyle`、`Transform`
- `crates/jy_media`
  - 负责媒体探测
  - 当前通过 `ffprobe` 读取视频/音频/图片的时长、尺寸和类型
- `crates/jy_timeline`
  - 负责把“素材 + 时间范围 + 变换”组装成 `Clip`
  - 提供 `ProjectBuilder`
- `crates/jy_draft`
  - 负责把 `Project` 转成剪映草稿 JSON 并写入草稿目录
- `crates/jy_template`
  - 负责模板模式
  - 可以加载已有草稿并执行素材替换、文本替换、草稿复制
- `crates/jy_cli`
  - 命令行入口
  - 所有对外命令都在这里暴露

## 3. 环境要求

运行前建议确认以下环境可用：

1. 已安装 Rust 和 Cargo
2. 已安装 `ffprobe`
3. 具备可写入剪映草稿目录的权限
4. 如果要用阿里 VOD JSON 转换，机器需要能访问 JSON 中的远程 `MediaURL`

### 3.1 检查 Rust

```powershell
rustc --version
cargo --version
```

### 3.2 检查 ffprobe

```powershell
ffprobe -version
```

如果 `ffprobe` 不在 PATH 中，`jy_media` 无法探测素材信息。

## 5. 命令总览

CLI 主入口在：

- `crates/jy_cli/src/main.rs`

当前可用命令包括：

- `init`
  - 生成一个空白 project manifest
- `generate`
  - 根据自定义 manifest 生成剪映草稿
- `generate-demo`
  - 根据本地视频、配音、BGM、SRT、水印直接生成 demo 草稿
- `import-bundle`
  - 将项目包导入为当前机器可打开的剪映草稿，支持 `timeline_package` 和 `draft_package`
- `vod-json-to-draft`
  - 将阿里云 VOD JSON 转换为剪映草稿
- `inspect`
  - 查看草稿中的轨道和素材信息
- `update`
  - 从 GitHub Releases 下载对应平台的发布包并更新本机 CLI
- `template-duplicate`
  - 复制一份模板草稿目录
- `template-replace-material-name`
  - 按素材名替换模板中的视频或音频素材
- `template-replace-material-seg`
  - 按轨道与片段位置替换模板中的素材
- `template-replace-text`
  - 替换模板中的文本或多段文本模板

### 5.1 输出模式

当前 CLI 支持两种输出模式：

- `text`
  - 默认模式
  - 适合人工在终端里直接执行和阅读
- `json`
  - 适合 Java / Go / Node 等后端调用
  - 输出稳定的单行 JSON 结果、错误和进度事件

全局参数：

```powershell
--output-format text
--output-format json
```

示例：

```powershell
cargo run -p jy_cli -- --output-format json generate `
  --project D:\temp\project.json `
  --output D:\JianyingPro Drafts\my_project
```

### 5.2 JSON 输出说明

如果使用 `--output-format json`：

1. 命令成功时会输出一行 `kind = result` 的 JSON
2. 命令失败时会输出一行 `kind = error` 的 JSON
3. 某些长任务会输出多行 `kind = progress` 的 JSON

以 `init` 为例，输出大致类似：

```json
{
  "kind": "result",
  "ok": true,
  "command": "init",
  "message": "Created project manifest: D:\\temp\\project.json",
  "data": {
    "manifest_path": "D:\\temp\\project.json",
    "name": "my_project",
    "canvas": {
      "width": 1920,
      "height": 1080,
      "fps": 30
    }
  }
}
```

失败时会包含：

- `code`
- `message`
- `command`

便于后端直接按字段判断，不需要依赖模糊的文本匹配。

### 5.3 更新 CLI

如果 CLI 是通过发布包或一条命令安装的，可以直接更新：

```powershell
jy update
```

指定版本：

```powershell
jy update --version v0.1.0
```

默认会更新当前安装目录。也可以显式指定：

```powershell
jy update --install-dir /path/to/YingDraft
```

macOS / Linux 会直接覆盖安装；Windows 会启动后台更新脚本，等当前 CLI 退出后再覆盖正在运行的可执行文件。

如果你准备让后端直接产出 `generate` 所需的 manifest JSON，请优先看：

- [Manifest规范.md](./Manifest规范.md)

如果你准备把当前 CLI 部署到服务端或交给 Java 后端调用，请优先看：

## 6. 常见使用方式

### 6.1 初始化 manifest

```powershell
cargo run -p jy_cli -- init `
  --name my_project `
  --width 1920 `
  --height 1080 `
  --fps 30 `
  --output D:\temp\project.json
```

这个命令适合你自己维护统一 manifest，然后再交给 `generate`。

### 6.2 根据 manifest 生成草稿

```powershell
cargo run -p jy_cli -- generate `
  --project D:\temp\project.json `
  --output D:\JianyingPro Drafts\my_project
```

这里的 `project.json` 必须符合 `jy_schema` 的结构要求。

### 6.3 生成本地 demo 草稿

这个命令适合快速验证：

- 主视频
- 配音
- BGM
- SRT 字幕
- 水印

```powershell
cargo run -p jy_cli -- generate-demo `
  --name rust_testVideo_demo `

  --bgm 'C:\Users\10740\Downloads\testVIdeo\方案1_bgm.wav' `

  --watermark 'C:\Users\10740\Downloads\testVIdeo\images.jpg' `
  --output 'D:\JianyingPro Drafts\rust_testVideo_demo'
```

实现位置：

- `crates/jy_cli/src/commands/generate_demo.rs`

### 6.4 将阿里 VOD JSON 转换为剪映草稿

这是目前最适合做“云端时间轴 -> 本地剪映草稿”的入口。

```powershell
cargo run -p jy_cli -- vod-json-to-draft `
  --config 'C:\Users\10740\Desktop\横屏.json' `
  --assets-dir 'D:\JianyingPro Drafts\vod_json_demo_assets' `
  --output 'D:\JianyingPro Drafts\vod_json_demo' `
  --name vod_json_demo
```

如果这条命令跑在阿里云同地域服务器上，可以打开内网下载：

```powershell
cargo run -p jy_cli -- vod-json-to-draft `
  --config 'C:\Users\10740\Desktop\横屏.json' `
  --assets-dir 'D:\JianyingPro Drafts\vod_json_demo_assets' `
  --output 'D:\JianyingPro Drafts\vod_json_demo' `
  --name vod_json_demo `
  --use-internal-url
```

`--use-internal-url` 只会改写阿里云 OSS/VOD 常见公网 Endpoint：

- `oss-cn-shanghai.aliyuncs.com`
- 改为 `oss-cn-shanghai-internal.aliyuncs.com`

如果 URL 已经是内网域名、加速域名或自定义域名，工具会保持原样。

这个命令会做四件事：

1. 读取阿里 VOD JSON
2. 下载 JSON 中的远程 `MediaURL`，可按参数走阿里云内网 Endpoint
3. 将 VOD 结构映射为 `Project / Track / Clip`
4. 输出为剪映草稿目录

如果你是后端调用，建议直接配合 JSON 输出模式：

```powershell
cargo run -p jy_cli -- --output-format json vod-json-to-draft `
  --config 'C:\Users\10740\Desktop\横屏.json' `
  --assets-dir 'D:\JianyingPro Drafts\vod_json_demo_assets' `
  --output 'D:\JianyingPro Drafts\vod_json_demo' `
  --name vod_json_demo
```

### 6.5 导入项目包并生成草稿

这个命令是“本地素材落地工具”的第一版 CLI 内核。

适合这样的场景：

1. 服务端生成项目包
2. 用户把项目包下载到自己的 Windows 或 macOS 机器
3. 本地 CLI 把素材引用解析为本机绝对路径
4. 最终生成剪映可直接打开的草稿

基础用法：

```powershell
cargo run -p jy_cli -- import-bundle `
  --source D:\Downloads\project_bundle.zip `
  --output D:\JianyingPro Drafts\bundle_demo `
  --name bundle_demo
```

其中：

- `--source`
  - 可以是已解压项目目录
  - 可以是 `.zip` 项目包
  - 也可以直接指向 `bundle.json`
- `--output`
  - 最终草稿目录
- `--name`
  - 可选，覆盖项目包中的草稿名

推荐优先阅读：

- [Bundle规范.md](./Bundle规范.md)

项目包目前支持两种模式：

1. `timeline_package`
   - 包里提供 `timeline.json`
   - 本地导入器重新生成草稿
2. `draft_package`
   - 包里提供现成 `draft/`
   - 本地导入器只重写素材绝对路径

`timeline_package` 的最小结构：

```text
project_bundle/
  bundle.json
  timeline.json
  assets/
    video/
    audio/
    image/
```

`bundle.json` 示例：

```json
{
  "bundle_version": 1,
  "project_id": "proj_001",
  "project_name": "bundle demo",
  "timeline_file": "timeline.json",
  "assets_dir": "assets"
}
```

`timeline.json` 示例：

```json
{
  "project": {
    "id": "proj_001",
    "name": "bundle demo"
  },
  "canvas": {
    "width": 1920,
    "height": 1080,
    "fps": 30
  },
  "assets": [
    {
      "id": "main_image",
      "kind": "image",
      "source": {
        "type": "bundle_path",
        "path": "image/poster.png"
      }
    }
  ],
  "tracks": [
    {
      "kind": "video",
      "name": "visual",
      "clips": [
        {
          "type": "image",
          "asset_id": "main_image",
          "timeline_in": 0,
          "timeline_out": 2000000
        }
      ]
    },
    {
      "kind": "text",
      "name": "caption",
      "clips": [
        {
          "type": "text",
          "timeline_in": "0s",
          "timeline_out": "2s",
          "text": "hello import bundle"
        }
      ]
    }
  ]
}
```

当前 `import-bundle` 已支持：

- `bundle_path`
- `local_path`
- `url`

以及下面四类片段：

- `video`
- `image`
- `audio`
- `text`

如果你的后端已经先用 `vod-json-to-draft` 生成了草稿，再把草稿和素材一起发给用户，请优先使用：

- `draft_package`

它的最小结构通常是：

```text
package_root/
  bundle.json
  draft/
    draft_content.json
    draft_info.json
    draft_meta_info.json
  assets/
```

最小示例：

```json
{
  "bundle_version": 1,
  "bundle_type": "draft_package",
  "project_id": "vod_demo_001",
  "project_name": "vod demo",
  "draft_dir": "draft",
  "assets_dir": "assets",
  "match_key": "name",
  "assets": [
    {
      "kind": "video",
      "match_value": "video_0001.mp4",
      "relative_path": "video_0001.mp4"
    }
  ]
}
```

这个模式下，命令会持续输出进度事件，当前已经覆盖：

- `resolve-assets`
  - 预扫描远程素材数量
- `download-asset`
  - 开始下载某个素材
- `save-asset`
  - 当前素材将落到哪个本地路径
- `download-progress`
  - 当前素材的字节进度 / 百分比
- `asset-finished`
  - 某个素材下载完成

文本模式下，终端里也会看到这些进度信息，只是表现为普通文本。

实现位置：

- `crates/jy_cli/src/commands/vod_json_to_draft.rs`

## 7. 阿里 VOD JSON 转换说明

### 7.1 当前已支持的字段

以你当前样例为准，已支持：

- `FECanvas.Width / Height`
- `OutputMediaConfig.Width / Height`
- `VideoTracks[].VideoTrackClips[]`
- `AudioTracks[].AudioTrackClips[]`
- `SubtitleTracks[].SubtitleTrackClips[]`
- 字幕中的：
  - `Content`
  - `TimelineIn`
  - `TimelineOut`
  - `FontSize`
  - `FontColor`
  - `FontColorOpacity`
  - `X`
  - `Y`
  - `Alignment`
  - `TextWidth`
  - `Outline`
  - `OutlineColour`
  - `FontFace`
  - `AdaptMode`
- 视觉覆盖层中的：
  - `Type = GlobalImage`
  - `MediaURL`
  - `X`
  - `Y`
  - `Width`
  - `Height`

### 7.2 当前转换规则

#### 视频轨

- 普通视频片段：
  - 根据 `TimelineIn / TimelineOut` 放到时间轴
  - 素材本身从开头截取，按目标时长截断
- 如果某个视频片段没有 `TimelineIn`：
  - 视为接在同轨前一个片段后面
- 如果某个视频片段没有 `TimelineOut`：
  - 视为使用素材默认时长

#### 音频轨

- 与视频轨同理
- 音频素材也会先下载到本地，再写入剪映绝对路径

#### 字幕轨

- 一条 `SubtitleTrack` 会被转成一条剪映文本轨
- 每个字幕片段会转成一个 `TextClip`
- `X / Y` 会直接映射到归一化坐标
- `FontSize` 会按当前实现缩放到 `TextStyle.size`

#### GlobalImage

- 当前作为“整段覆盖层”处理
- 会生成一个视觉轨道中的图片/动图片段
- `X / Y / Width / Height` 用像素坐标推导为剪映的 `Transform`

### 7.3 当前已知限制

这部分很重要，后续二开时建议优先看：

1. `AdaptMode` 目前只做了最基础处理
   - `Contain` 现在基本等价于不额外裁剪
   - 还没有完整映射到更复杂的剪映画面填充行为
2. `GlobalImage` 的几何映射是第一版近似实现
   - 对复杂 GIF/PNG 覆盖层已经可用
   - 但不保证和阿里预览逐像素一致
3. 字体名当前没有映射到剪映字体资源
   - 目前保留的是颜色、对齐、描边等基础样式
4. 远程素材会先全部下载到本地
   - 这是因为剪映草稿必须引用本机绝对路径
5. 目前没有做更深层的 VOD 特效、转场、滤镜映射

## 8. 二次开发建议

如果你后面要继续自己开发，推荐按下面的层次去改：

### 8.1 想改输入格式

比如你要新增：

- 自定义 JSON
- 数据库配置
- Web 前端提交的数据

优先改这里：

- `crates/jy_cli/src/commands/...`

思路是：

1. 先把外部输入解析成中间结构
2. 再转换成 `jy_schema::Project`
3. 最后调用 `write_draft`

### 8.2 想改时间轴拼接规则

优先看：

- `crates/jy_timeline/src/builder.rs`
- `crates/jy_timeline/src/clip.rs`

这里负责：

- 创建轨道
- 创建视频/音频/文本/图片片段
- 做时间范围和速度换算
- 做同轨重叠校验

### 8.3 想改草稿 JSON 结构

优先看：

- `crates/jy_draft/src/converter.rs`

这里是：

- `Project -> draft_content.json` 的核心转换层

如果剪映打开后效果不对，通常都要从这里查。

### 8.4 想增强模板替换

优先看：

- `crates/jy_template/src/lib.rs`

这里负责：

- 加载现有草稿
- 替换视频/音频素材
- 替换单段文本或多段文本模板
- 复制模板草稿目录

### 8.5 想增强阿里 VOD 转换

优先看：

- `crates/jy_cli/src/commands/vod_json_to_draft.rs`

建议的演进顺序：

1. 完善 `AdaptMode`
2. 做字体资源映射
3. 做 `GlobalImage` 更精确的位置/缩放换算
4. 对素材做 URL 级别去重和缓存复用
5. 继续补特效、滤镜、转场

## 9. 推荐开发流程

建议每次新增能力都按这个顺序走：

1. 在 `jy_schema` 明确数据结构
2. 在 `jy_timeline` 实现 clip/build 逻辑
3. 在 `jy_draft` 确认导出 JSON 是否能被剪映识别
4. 在 `jy_cli` 增加一个可独立验证的命令
5. 用真实素材在剪映中打开验证

## 10. 源码入口地图

这一节适合你后面自己开发时快速定位文件。

### 10.1 如果你想新增一种输入格式

比如新增：

- 另一个平台导出的 JSON
- 数据库存储的时间轴
- Web 前端提交的 timeline payload

优先从这里开始：

- `crates/jy_cli/src/commands/`

建议流程：

1. 新建一个命令文件
2. 先定义输入结构
3. 把输入结构映射成 `jy_schema::Project`
4. 最后调用 `jy_draft::writer::write_draft`

### 10.2 如果你想修改模板替换规则

优先看：

- `crates/jy_template/src/lib.rs`

这个文件是模板模式的核心，主要负责：

- 选轨道
- 找片段
- 替换素材
- 替换文本
- 处理时间范围伸缩

如果你想改“替换后片段应该怎么变长/变短”，重点看：

- `process_timerange`
- `ShrinkMode`
- `ExtendMode`

### 10.3 如果你想修改草稿导出 JSON

优先看：

- `crates/jy_draft/src/converter.rs`
- `crates/jy_draft/src/writer.rs`

职责区分：

- `converter.rs`
  - 负责 `Project -> draft_content.json`
- `writer.rs`
  - 负责落盘，生成整个草稿目录

如果剪映里“能打开但效果不对”，一般优先排查 `converter.rs`。

### 10.4 如果你想改时间轴拼接逻辑

优先看：

- `crates/jy_timeline/src/builder.rs`
- `crates/jy_timeline/src/clip.rs`

这里负责：

- 新建轨道
- 创建 clip
- 处理 source/target 时间范围
- 校验片段重叠
- 维护总时长

### 10.5 如果你想改坐标与画布映射

优先看：

- `crates/jy_schema/src/transform.rs`
- `crates/jy_cli/src/commands/vod_json_to_draft.rs`

特别是：

- Web 预览与剪映坐标对齐
- 阿里 VOD 中 `X / Y / Width / Height` 到剪映 `Transform` 的映射
- GlobalImage 覆盖层位置换算

### 10.6 如果你想改媒体探测或素材路径规则

优先看：

- `crates/jy_media/src/probe.rs`
- `crates/jy_media/src/material.rs`

这层负责：

- 调 `ffprobe`
- 判断是视频/图片/音频
- 获取时长和尺寸
- 统一转成本机绝对路径

如果后面你想加：

- 远程素材缓存策略
- 本地素材哈希去重
- 图片/GIF 特殊处理

基本都从这层开始。

## 11. 常见问题

### 11.0 为什么现在推荐优先使用 `--output-format json` 给后端接入？

因为当前 CLI 已经补了统一的结构化输出，后端接入时可以直接得到：

- 稳定的成功结果结构
- 稳定的错误结构
- VOD 下载进度事件

这样 Java / Go / Node 在集成时不需要再靠 `stdout` 文本做脆弱解析。

### 11.1 为什么草稿能生成，但剪映里打不开素材？

最常见原因是素材路径不对。

剪映草稿依赖本机绝对路径，所以：

- 本地素材必须存在
- 下载后的素材不能被移动
- 如果换机器，草稿里的路径也要重写

### 11.2 为什么需要先下载阿里 VOD 的远程素材？

因为剪映不会直接使用 HTTP URL 作为素材路径。  
所以必须先把远程资源下载到本机，再把本机绝对路径写入草稿。

### 11.3 为什么视频时长会有 1ms 误差？

远程 JSON 里的秒值通常是浮点数，而素材真实时长来自 `ffprobe`。  
二者可能存在 1ms 左右差异，所以当前实现会在必要时自动截断 source range，避免超出素材真实时长。

## 12. 当前建议

如果你后面准备继续把这个工程做成正式产品，建议优先把下面三项做稳：

1. 统一 timeline schema
2. 阿里 VOD JSON 到 schema 的稳定映射
3. 剪映预览位置与 Web 预览位置的统一坐标系

这样以后无论你接：

- 前端预览
- 批量短剧解说生成
- 模板替换
- Tauri 本地落地

都可以复用同一套内核。
