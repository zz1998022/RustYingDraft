# `YingDraft` 部署文档

## 1. 文档目的

本文档面向准备把 `YingDraft` 放到真实环境中使用的人，重点回答这些问题：

1. 当前项目推荐怎么部署
2. Java 后端怎么调用
3. `ffprobe` 依赖怎么统一管理
4. 哪些能力适合放服务端，哪些不适合
5. Windows、macOS、Linux 分别要注意什么

这份文档会优先结合你当前的实际场景：

- 主业务仍然是 Web
- 后端可能是 Java
- 当前暂不整体迁移到 Tauri
- 剪映草稿依赖本机绝对路径
- 后续可能会有本地 companion app

## 2. 当前项目的部署定位

当前 `YingDraft` 更适合被部署成下面两种形态。

### 2.1 CLI 工具

这是当前最成熟、最直接的部署方式。

形态：

- 编译出 `jy_cli` 可执行文件
- 由外部系统通过命令行调用

适合：

- Java 后端通过 `ProcessBuilder` 调用
- 本地脚本批处理
- 运维任务 / 定时任务
- 桌面应用内部调用

### 2.2 被上层系统复用的 Rust 内核

也就是把这些 crate 当内部库继续封装：

- `jy_schema`
- `jy_timeline`
- `jy_draft`
- `jy_template`
- `jy_media`

适合：

- 后续做 HTTP 服务
- 后续做 Tauri companion app
- 做你自己的 Rust 上层业务模块

## 3. 推荐的部署结论

如果你现在要尽快上线，最推荐的是：

1. 服务端部署 `jy_cli`
2. 服务端同时部署 `ffprobe`
3. Java 后端通过命令行调用 `jy_cli`
4. 服务端主要负责：
   - manifest 生成
   - 阿里 VOD JSON 转换
   - 模板替换
   - 项目包生成
5. 用户最终要在自己电脑里打开的剪映草稿，不建议直接由服务端最终产出

原因很关键：

- 剪映草稿中的素材路径必须是生成草稿那台机器上的绝对路径
- 服务端生成出来的是服务端路径
- 用户电脑上通常不能直接复用这些路径

所以在产品层面，更合理的方案是：

1. 服务端生成：
   - timeline
   - manifest
   - 项目包
2. 本地端生成：
   - 最终可打开的剪映草稿

## 4. 当前可部署的能力边界

### 4.1 适合部署到服务端的能力

当前适合放服务端的能力：

1. 时间轴转换
2. 阿里 VOD JSON 转内部项目结构
3. manifest 生成
4. 模板替换
5. 项目包生成
6. 草稿预生成

这里的“草稿预生成”指的是：

- 草稿和素材都只在服务端内部使用
- 或者只是为了测试、验收、内部处理

### 4.2 不适合直接作为最终用户交付的能力

如果用户最终要在自己的电脑上打开剪映草稿，服务端**不适合**直接生成最终草稿交付。

原因：

- 草稿里的素材路径是服务端绝对路径
- 用户电脑没有这些路径
- 剪映打开后会出现素材丢失

如果你的目标是“让用户下载后直接导入剪映”，请改成：

1. 服务端打项目包
2. 本地工具重写绝对路径
3. 本地工具生成最终草稿

## 5. 当前运行依赖

部署当前代码时，至少要满足这些运行条件：

1. 操作系统具备文件读写权限
2. 目标环境可执行 `jy_cli`
3. 目标环境可执行 `ffprobe`
4. 如果用 `vod-json-to-draft`，目标环境需要能访问阿里 VOD JSON 中的远程 `MediaURL`

### 5.1 Rust 是否必须装在目标机器上

不必须。

如果你已经在构建机上编译好了发布产物，那么目标机器只需要：

- `jy_cli`
- `ffprobe`
- 运行时依赖的系统环境

也就是说：

- 构建机需要 Rust
- 运行机不一定需要 Rust

### 5.2 为什么需要 `ffprobe`

当前媒体探测逻辑在：

- [probe.rs](./crates/jy_media/src/probe.rs)

当前实现通过外部命令调用：

- `ffprobe`

它用于读取：

- 视频时长
- 音频时长
- 视频宽高
- 媒体类型

### 5.3 当前代码的真实情况

当前代码**还没有**内置 `ffprobe` 路径解析器。

这意味着现在默认行为是：

- 从系统 `PATH` 中查找 `ffprobe`

所以现实中的部署方法有两种：

1. 目标机器提前安装 `ffprobe`
2. 你把 `ffprobe` 跟 `jy_cli` 一起打包分发，并在启动前把它所在目录加入 `PATH`

## 6. 推荐目录结构

### 6.1 服务端部署目录

Windows 示例：

```text
D:\deploy\YingDraft\
  bin\
    jy_cli.exe
    ffprobe.exe
  work\
  logs\
  scripts\
```

Linux / macOS 示例：

```text
/opt/YingDraft/
  bin/
    jy_cli
    ffprobe
  work/
  logs/
  scripts/
```

说明：

- `bin/`：可执行文件
- `work/`：每次任务的工作目录
- `logs/`：运行日志
- `scripts/`：启动脚本、清理脚本、巡检脚本

### 6.2 每个任务的工作目录

建议每个任务都使用独立目录，避免并发互相覆盖。

示例：

```text
work/
  task_20260422_001/
    input/
    assets/
    output/
    logs/
```

可以约定：

- `input/`
  - 放 manifest、阿里 VOD JSON、模板配置等输入文件
- `assets/`
  - 放下载后的媒体文件
- `output/`
  - 放生成的草稿目录或中间结果
- `logs/`
  - 放本次任务的命令日志

## 7. 构建与发布

进入工程目录：

```powershell
cd /path/to/YingDraft
```

### 7.1 Debug 构建

```powershell
cargo build -p jy_cli
```

产物通常在：

```text
target/debug/jy_cli(.exe)
```

### 7.2 Release 构建

生产环境建议使用：

```powershell
cargo build -p jy_cli --release
```

产物通常在：

```text
target/release/jy_cli(.exe)
```

### 7.3 发布建议

建议发布时至少准备这些内容：

1. `jy_cli` 可执行文件
2. `ffprobe`
3. 一份启动脚本
4. 一份环境检查脚本
5. 当前文档和主使用文档

## 8. `ffprobe` 管理方案

### 8.1 方案 A：系统全局安装

优点：

- 最简单

缺点：

- 环境依赖不够可控
- 机器迁移时容易漏装

适合：

- 内部机器
- 你自己完全控制的服务器

### 8.2 方案 B：跟程序一起分发

这是当前更推荐的方案。

虽然代码里还没有“优先找内置 `ffprobe`”的逻辑，但部署层完全可以先解决这个问题。

做法：

1. 把 `ffprobe` 放到 `bin/`
2. 启动 `jy_cli` 前先把 `bin/` 加到 `PATH`
3. 再调用 `jy_cli`

Windows 示例：

```powershell
$env:PATH = "D:\deploy\YingDraft\bin;$env:PATH"
D:\deploy\YingDraft\bin\jy_cli.exe generate --project D:\work\project.json --output D:\work\draft_out
```

Linux / macOS 示例：

```bash
export PATH="/opt/YingDraft/bin:$PATH"
/opt/YingDraft/bin/jy_cli generate --project /data/work/project.json --output /data/work/draft_out
```

### 8.3 为什么当前推荐用部署层解决

因为你现在的目标是尽快上线，而不是先做一轮依赖管理重构。

所以当前最现实的统一管理方式是：

- 把 `ffprobe` 和 `jy_cli` 一起部署
- 由启动脚本统一配置 `PATH`

这样已经能做到“统一打包、统一发布、统一调用”。

### 8.4 后续可优化方向

如果后面你要继续增强部署体验，建议补一个 `ffprobe` 解析器，查找顺序可以设计成：

1. 程序内置目录
2. 环境变量指定路径
3. 系统 `PATH`

但这不是当前上线前的硬阻塞。

## 9. Java 后端接入方式

当前最推荐的是 Java 后端通过命令行调用 `jy_cli`。

### 9.1 调用思路

Java 后端负责：

1. 准备输入文件
2. 选择工作目录
3. 调用 `jy_cli`
4. 读取退出码
5. 读取标准输出与错误输出
6. 收集结果文件

### 9.2 最适合 Java 接入的命令

当前最适合：

- `generate`
- `vod-json-to-draft`
- `template-duplicate`
- `template-replace-material-name`
- `template-replace-material-seg`
- `template-replace-text`

### 9.3 示例：生成草稿

```powershell
jy_cli generate --project D:\work\project.json --output D:\work\draft_out
```

### 9.4 示例：阿里 VOD JSON 转草稿

```powershell
jy_cli vod-json-to-draft --config D:\work\vod.json --assets-dir D:\work\assets --output D:\work\draft_out --name my_draft
```

### 9.5 Java `ProcessBuilder` 示例

```java
ProcessBuilder pb = new ProcessBuilder(
    "D:\\deploy\\YingDraft\\bin\\jy_cli.exe",
    "generate",
    "--project", "D:\\deploy\\YingDraft\\work\\task_001\\input\\project.json",
    "--output", "D:\\deploy\\YingDraft\\work\\task_001\\output\\draft_out"
);

pb.environment().put(
    "PATH",
    "D:\\deploy\\YingDraft\\bin;" + pb.environment().getOrDefault("PATH", "")
);

pb.redirectErrorStream(true);
Process process = pb.start();

String output = new String(
    process.getInputStream().readAllBytes(),
    java.nio.charset.StandardCharsets.UTF_8
);

int exitCode = process.waitFor();
if (exitCode != 0) {
    throw new RuntimeException("jy_cli failed: " + output);
}

System.out.println(output);
```

### 9.6 Java 侧建议补的封装

Java 后端最好统一封装这些能力：

1. 任务目录初始化
2. 输入文件写入
3. 命令执行
4. 输出日志采集
5. 超时控制
6. 失败重试或失败归档

## 10. 常见部署形态

### 10.1 形态 A：纯服务端内部使用

适用场景：

- 草稿只在服务端内部消费
- 内部流水线验证
- 测试环境

特点：

- 最简单
- 不涉及最终用户本机路径问题

### 10.2 形态 B：服务端生成项目包

适用场景：

- 用户最终要在自己电脑上打开剪映草稿

特点：

- 服务端只负责：
  - timeline
  - manifest
  - 远程资源整理
  - 项目包
- 本地工具负责：
  - 绝对路径落地
  - 最终草稿生成

这是后续更合理的产品化方向。

### 10.3 形态 C：桌面 companion app 内置 Rust 内核

适用场景：

- 你后面做跨平台本地导入器
- 需要图形化
- 需要解决最终路径问题

当前这份文档不展开实现细节，具体设计见：

- [COMPANION_APP_DESIGN.md](./COMPANION_APP_DESIGN.md)

## 11. Windows / macOS / Linux 说明

### 11.1 代码层面是否跨平台

从 Rust 代码本身看，当前主干逻辑基本是跨平台的：

- `jy_schema`
- `jy_timeline`
- `jy_draft`
- `jy_template`
- `jy_cli`

都可以跨平台编译。

### 11.2 实际业务层面限制

虽然代码基本跨平台，但业务上要分情况：

#### Windows

- 当前最适合部署
- 对剪映用户最常见
- `jy_cli + ffprobe.exe` 组合最容易部署

#### macOS

- 代码可以运行
- 也适合后续做本地 companion app
- 但如果最终目标是“用户在 mac 上用剪映打开草稿”，仍需要你继续验证目录、版本和兼容性

#### Linux

- 代码可以运行
- 做服务端处理完全可行
- 但 Linux 通常不是最终打开剪映草稿的平台

所以可以这样理解：

- 做服务端处理：Windows / macOS / Linux 都可以
- 做最终用户草稿落地：重点还是 Windows / macOS

## 12. 当前已知风险点

### 12.1 `ffprobe` 未找到

现象：

- 媒体探测失败
- 命令报错退出

解决：

- 确保 `ffprobe` 已安装，或已经随部署包加入 `PATH`

### 12.2 输入路径不是绝对路径

现象：

- 草稿生成成功，但剪映里素材丢失

解决：

- 尽量保证输入素材是绝对路径
- 如果是服务端下载素材，下载后应使用本机绝对路径再写入

### 12.3 服务端直接生成用户最终草稿

现象：

- 用户下载后打开草稿，素材全部丢失

原因：

- 素材路径是服务端路径，不是用户本机路径

解决：

- 改成“服务端项目包 + 本地生成最终草稿”

### 12.4 远程资源不可访问

现象：

- `vod-json-to-draft` 下载素材失败

解决：

- 确保服务器能访问阿里 VOD JSON 中的远程地址
- 必要时在 Java 层提前校验 URL 连通性

## 13. 上线前检查清单

建议上线前至少检查一遍下面这些项：

1. `jy_cli` 可执行文件能正常运行
2. `ffprobe` 可执行文件能在当前环境找到
3. Java 调用脚本已把 `bin/` 加入 `PATH`
4. 工作目录具备写权限
5. 日志目录具备写权限
6. 每个任务都有独立目录
7. 输入素材路径使用绝对路径
8. 如果用 `vod-json-to-draft`，目标环境可以访问远程资源
9. 错误输出已被 Java 后端完整采集
10. 至少用一组真实素材在目标环境中跑通过一次

## 14. 当前最实用的部署建议

如果你现在要尽快上线，我建议直接按这套方式部署：

### 服务端

- 部署 `jy_cli`
- 部署 `ffprobe`
- 用 Java 调 CLI

服务端适合做：

- manifest 处理
- 阿里 VOD JSON 转换
- 模板替换
- 项目包生成

### 用户侧

不要让服务端直接交付“最终可打开草稿”给用户。

更合理的是：

1. 服务端生成项目包 / timeline / manifest
2. 本地工具根据本机绝对路径生成最终草稿

## 15. 后续优化建议

从部署角度，后续最值得做的增强有两个：

1. 给 `jy_media` 加 `ffprobe` 路径解析器
   - 优先找程序内置目录
   - 再找环境变量
   - 最后找系统 `PATH`
2. 给 `jy_cli` 增加更稳定的结构化输出
   - 例如 JSON 结果
   - 方便 Java 后端解析

但在你当前阶段，这两件事都不是上线前的硬阻塞。
